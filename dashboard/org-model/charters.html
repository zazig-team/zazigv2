Charters: Mandates & Interdictions

A charter is the governance contract for a worker. Mandates define jurisdiction. Interdictions define constitutional constraints. Together they prevent authority collisions in an autonomous org.

Why Charters Exist

When 5+ workers operate independently, overlapping mandates create conflicts and missing interdictions create overreach. Charters are the separation of powers.

Without Charters

  • CPO and CTO both claim architecture decisions
  • VP-Eng dispatches agents without budget checks
  • Senior engineers modify infrastructure unilaterally
  • Nobody owns QA — reviews get skipped

With Charters

  • Architecture is CTO's mandate, CPO's interdiction
  • VP-Eng has dispatch mandate, bounded by budget hooks
  • Infra changes require CTO approval (enforced, not suggested)
  • QA is VP-Eng's mandate — no ambiguity

Why Separate from Role Prompts

Charters could live inside role prompts, but separating them into a dedicated governance layer provides three structural advantages.

1. Charters are relational. CPO's interdiction "NEVER make architecture decisions" exists because CTO's mandate says "owns architecture." A role prompt is self-contained; a charter is defined by its relationship to other charters.

2. Charters are cross-validatable. The orchestrator can detect mandate overlaps and interdiction gaps at config time. If two execs both claim "owns security posture," that's flagged before any agent runs. Role prompts can't be cross-validated because they're freeform text.

3. Charters are enforceable. A charter interdiction stored in the orchestrator can be enforced (blocked at the hooks layer), not just suggested. A role prompt saying "don't write code" is a suggestion. An interdiction in the orchestrator is a wall.

Charter by Tier

Not every worker needs the same governance weight. Charter scope scales with autonomy.

Tier Charter Rationale
Executives Full — mandates + interdictions Autonomous actors with broad scope. Need explicit jurisdiction boundaries.
Employees Light — mainly interdictions Narrow scope. Mandates are implicit in role prompt. Interdictions prevent scope creep.
Contractors None No autonomy. Task-scoped by definition. No governance needed beyond the job spec.

Example Charters

Each executive has a full charter defining what they own (mandates) and what they must never do (interdictions). Employees get interdictions only.

CPO Charter

charter: cpo 
{
  "role": "CPO",
  "tier": "executive",

  "mandates": [
    "Product strategy ownership",
    "Card prioritization",
    "Design doc authorship",
    "Standup synthesis",
    "Code review decisions",
    "Founder communication",
    "Directive authorship"
  ],

  "interdictions": [
    "NEVER dispatch agents",
    "NEVER write code",
    "NEVER make architecture decisions",
    "NEVER override CTO"
  ]
}

CTO Charter

charter: cto 
{
  "role": "CTO",
  "tier": "executive",

  "mandates": [
    "Architecture decisions",
    "Engineering standards",
    "Security posture",
    "Tech review gate",
    "DX ownership"
  ],

  "interdictions": [
    "NEVER make product decisions",
    "NEVER dispatch agents",
    "NEVER write production code"
  ]
}

VP-Eng Charter

charter: vp-eng 
{
  "role": "VP-Eng",
  "tier": "executive",

  "mandates": [
    "Execution (pull cards, write specs, dispatch agents)",
    "QA coordination",
    "Card lifecycle",
    "Report synthesis"
  ],

  "interdictions": [
    "NEVER make product/architecture decisions",
    "NEVER add team label",
    "NEVER skip QA"
  ]
}

Senior Engineer Charter

charter: senior-engineer (employee tier) 
{
  "role": "Senior Engineer",
  "tier": "employee",

  "mandates": [],

  "interdictions": [
    "NEVER merge without review",
    "NEVER modify infra",
    "NEVER change schemas without CTO approval"
  ]
}

Note the Senior Engineer has empty mandates. For employees, mandates are implicit in the role prompt. Charters only add explicit interdictions to prevent scope creep beyond the role's natural boundary.

Governance Validation

At startup, the orchestrator runs three validation checks across all loaded charters. Violations are surfaced to founders — never silently ignored.

Check 1
Mandate Overlap
Flag if two executives claim the same jurisdiction. Example: both CPO and CTO list "security posture" as a mandate.
Check 2
Interdiction Coverage
Flag if a high-impact action appears in no executive's interdictions. Example: "deploy to production" isn't forbidden for anyone.
Check 3
Mandate-Interdiction Consistency
Flag if a role's mandate conflicts with its own or another role's interdiction. Example: VP-Eng mandated to dispatch but also interdicted from dispatch.

Validate

Pass

Fail

System can start in degraded mode, but violations require human resolution

Governance violations are never silently swallowed. A mandate overlap might be intentional (shared responsibility), but it must be explicitly acknowledged by a founder before the system runs clean.

Charter Evolution

Charters are not static. They evolve as the org adds roles, adjusts scope, and learns from operational experience. Two primary flows.

New Executive Role

Step 1
Draft Charter
Define mandates and interdictions for the new role
Step 2
Cross-Validate
Check new charter against all existing charters for overlaps and gaps
Step 3
Resolve Conflicts
Adjust mandates and interdictions across affected roles
Step 4
Founder Approves
Human sign-off before the charter takes effect

Charter Amendment

Atomic updates: When amending a charter, both the amended charter and any affected charters must be updated atomically. You cannot remove a mandate from one role without adding it to another — or explicitly declaring the action unowned.

Adding a Mandate

  • Check no other exec already claims it
  • If claimed, negotiate transfer or shared ownership
  • Add corresponding interdiction to other relevant roles
  • Re-run governance validation

Removing a Mandate

  • Identify which role inherits the mandate
  • Transfer mandate to new owner
  • Update interdictions across affected roles
  • Re-run governance validation
