# Spec: Sync WebUI and Edge Function Production Deployments

**Idea ID:** 8f6a5718-ff41-4d0d-a5d9-32e2cf502472
**Priority:** High
**Domain:** Engineering
**Complexity:** Simple
**Date:** 2026-03-19

---

## Problem

Vercel auto-deploys the WebUI to **production** on every push to `master`. Supabase edge functions, however, only deploy to **staging** on `master` push ŌĆö they only reach production when `zazig promote` runs (which fast-forwards the `production` branch).

This creates a broken window in production:

```
master push
  ŌåÆ WebUI ships to production immediately (Vercel Git integration)
  ŌåÆ Edge functions ship to staging only (deploy-edge-functions.yml)
  ŌåÆ Production edge functions are stale until zazig promote runs
```

New features that touch both WebUI and edge functions are live on the frontend but calling nonexistent or outdated functions in production, producing "Load failed" errors. The job detail inline expand feature was a confirmed victim.

---

## Current Deployment Model

| Artifact | Trigger | Environment |
|---|---|---|
| WebUI (Vercel) | push to `master` | **Production** (broken) |
| Edge functions | push to `master` | Staging only |
| Edge functions | push to `production` | Production |
| DB migrations | push to `production` | Production |
| CLI + Agent binaries | `zazig promote` run locally | Production |
| `agent_versions` row | `zazig promote` run locally | Production |

The `production` branch is only updated via `zazig promote`, which:
1. Builds and bundles CLI + agent binaries
2. Bumps package versions
3. Fast-forwards `production` from `master`
4. Pushes `production`, triggering `deploy-production.yml` (migrations ŌåÆ edge functions)
5. Registers `agent_versions` in Supabase
6. Creates a GitHub Release with binaries

---

## Recommended Solution: Option 1 ŌĆö Align Vercel to Deploy from `production` Branch

Change Vercel's production environment to watch the `production` branch instead of `master`.

**Why this is the right fix:**
- The `production` branch already gates all other production artifacts (edge functions, migrations, binaries, agent_versions). Making WebUI consistent with this pattern requires a single config change.
- `zazig promote` already pushes `production` after running all pre-flight checks ŌĆö adding WebUI to that push makes the entire release atomic.
- No code changes required, just a Vercel dashboard setting.
- Vercel preview deployments continue to work on PRs/`master` as normal.

**Why not Option 2 (auto-promote edge functions on merge):**
- Edge function deployment to production requires DB migrations to run first (`supabase db push` in `deploy-production.yml`). Bypassing promote would risk deploying functions against a schema that hasn't been migrated.
- The promote gate is intentional and tied to binary compilation, versioning, and GitHub Releases. Partially automating production bypasses this safety.

**Why not Option 3 (WebUI to staging only):**
- Requires a second Vercel project for staging (new domain, env vars, routing), which is more setup than needed.
- Option 1 achieves the same synchronisation with less infrastructure change.

## Implementation

### Step 1: Change Vercel Production Branch (Human Task)

In the Vercel dashboard for the zazigv2 project:
1. Navigate to **Project Settings ŌåÆ Git**
2. Change **Production Branch** from `master` to `production`
3. Save

This single change makes Vercel production deployments happen only when `zazig promote` runs (the same trigger as edge functions and migrations).

### Step 2: Verify Preview Deployments Still Work

Vercel preview deployments should continue to trigger on non-production branches (PRs, `master`). After changing the production branch, confirm:
- PRs still generate preview URLs
- `master` pushes trigger a preview deployment (not production)

### Step 3: Document the Release Model

Update any developer documentation or README sections that describe the deploy process to reflect the new model:
- `master` push ŌåÆ CI, tests, staging deploy (edge functions + WebUI preview)
- `zazig promote` ŌåÆ full production deploy (migrations + edge functions + WebUI + binaries)

## Acceptance Criteria

1. After merging a feature branch to `master`, the **production** WebUI at the Vercel production URL does **not** update.
2. After merging a feature branch to `master`, a Vercel **preview** deployment is created for `master`.
3. After running `zazig promote`, the production WebUI **does** update within the same deployment window as edge functions and DB migrations.
4. A feature that adds a new edge function and a new WebUI component, when released via `zazig promote`, has both pieces live simultaneously in production ŌĆö no broken window.
5. The `production` branch in GitHub triggers all three production artifacts in sequence: DB migrations ŌåÆ edge functions (via `deploy-production.yml`) ŌåÆ WebUI (via Vercel).
6. No regressions to PR preview deployments on Vercel.

## Complexity Estimate

**Simple** ŌĆö single Vercel dashboard config change. No code modifications. No new infrastructure. The `production` branch trigger already exists and is fully functional for edge functions and migrations. This is a configuration alignment, not a new feature.

---

## Risks and Considerations

- **Slower feedback in production**: Developers used to seeing `master` changes live immediately in the production WebUI will now need to wait for `zazig promote`. This is intentional ŌĆö it's the correct release gate.
- **Hotfixes**: For urgent production fixes, `zazig promote` must still be run. This is already the case for edge functions and migrations, so no change in operational process.
- **Vercel preview on `master`**: The `master` preview URL can serve as a staging WebUI to validate changes before promote. Consider bookmarking/sharing this URL with the team as a staging preview.
