# Automated QA System Design

**Date:** 2026-03-25
**Status:** Approved for implementation
**Authors:** Tom Weaver, Claude

---

## Problem

Chris spends significant time manually testing staging before every promote. There is no automated way to know if staging is healthy, no mechanism that catches regressions close to when they're introduced, and no connection between what was built and a definition of what "done" looks like. The result is a human bottleneck on every release cycle.

---

## Solution Overview

Four components, built in sequence:

1. **Spec Worker** — writes acceptance criteria at feature creation time, before breakdown
2. **Zazig Workers QA Runner** — external Go binary, runs probes hourly against staging, reports to Supabase
3. **Failure Response** — converts failures to bug ideas, attempts auto-fix for known issues, Slacks on critical failures
4. **Promote Gate** — `zazig promote` checks recent QA results before proceeding

The continuous runner (not a merge gate, not a promote gate) provides ambient health signal. By the time a promote happens, you already know if staging is green.

## Phase 1: Foundation — DB Schema + Spec Worker

**Where:** zazig repo
**When:** now

### 1a. DB Schema

Two additions to the existing schema.

**`features.acceptance_criteria`** — JSONB column storing structured test cases generated by the spec worker:

```sql
ALTER TABLE features ADD COLUMN acceptance_criteria JSONB DEFAULT '[]'::jsonb;
```

Each entry in the array follows this shape:

```json
{
  "id": "tc-001",
  "name": "Reject button appears on triaged idea",
  "type": "ui | api | db | probe | code-review",
  "severity": "critical | high | medium | low",
  "description": "Human-readable description of what is being verified",
  "steps": ["step 1", "step 2"],
  "expected": "What a passing result looks like",
  "tools": ["playwright", "supabase-rest"]
}
```

**`qa_results`** — stores each probe run outcome:

```sql
CREATE TABLE qa_results (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  feature_id UUID REFERENCES features(id),       -- null for health checks not tied to a feature
  test_case_id TEXT NOT NULL,                     -- matches acceptance_criteria[].id, or hardcoded health check id
  run_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  status TEXT NOT NULL CHECK (status IN ('pass', 'fail', 'error', 'skip')),
  details JSONB,                                  -- what was checked, what was found, logs
  auto_fixed BOOLEAN NOT NULL DEFAULT false,
  bug_idea_id UUID REFERENCES ideas(id),          -- set when a bug idea is created from this failure
  runner_version TEXT,
  environment TEXT NOT NULL DEFAULT 'staging',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX qa_results_env_run_at_idx ON qa_results(environment, run_at DESC);
CREATE INDEX qa_results_status_idx ON qa_results(environment, status, run_at DESC);
```

### 1b. Spec Worker

A new worker role that fires when a feature is created, before breakdown.

**Trigger:** feature status transitions to `created` (or equivalently, feature is first inserted)

**Pipeline position:** runs in parallel with or just before the breakdown specialist. The breakdown specialist should wait for acceptance criteria to be written so it can reference them when defining job success conditions.

**Prompt behaviour:**
- Receives: feature title, description, project context, recent changes to the codebase (if available)
- Produces: structured JSON array of test cases covering the feature's acceptance criteria
- Writes back to `features.acceptance_criteria` via MCP tool

**Test case generation guidelines the worker should follow:**
- Cover the happy path first (does the feature work as described?)
- Cover obvious failure modes (what should happen when it doesn't?)
- Assign severity: `critical` for anything that blocks core workflow, `high` for primary feature paths, `medium` for edge cases
- Choose the simplest probe type that verifies the criterion (prefer `db` over `ui` where both would work)
- For UI changes, include a `ui` test case with Playwright steps
- For every feature, include at least one `code-review` test case asking Claude to verify the implementation matches the spec

**Example output for "Add Reject button to triaged ideas":**

```json
[
```

## Phase 2: Zazig Workers QA Runner

**Where:** Zazig Workers repo (external, independent of zazig daemon)
**When:** after Phase 1 is in place

### Brief for implementation

---

### QA Runner — Implementation Brief

**Context**

zazig is a platform that orchestrates AI agents (Claude Code, Codex) to build software features autonomously. It has three layers: a Supabase backend (Postgres + Realtime + Edge Functions), a local agent daemon (Node.js, runs on contributor machines), and a web dashboard (React). The staging environment runs against a separate Supabase project but uses the same codebase.

This QA runner is a new worker type in Zazig Workers. It runs on a schedule, independently of the zazig daemon, and verifies that staging is working correctly. It reads test cases from the zazig Supabase DB and writes results back to it.

**Core responsibility**

Every hour, run a full suite of probes against the staging environment. Each probe checks one thing. Write the result to `qa_results`. If something fails, attempt an auto-fix or create a bug idea. Never depend on the zazig daemon being healthy — this runner must work even when the daemon is broken.

**Supabase connection**

- Connect directly to the staging Supabase project using the service role key
- All reads and writes go through the Supabase REST API or Postgres client
- Env vars required:
  - `SUPABASE_STAGING_URL` — the staging project URL
  - `SUPABASE_STAGING_SERVICE_ROLE_KEY` — service role key (full access)
  - `ANTHROPIC_API_KEY` — for code-review probe type
  - `SLACK_WEBHOOK_URL` — for critical failure notifications

**What to test**

Two categories:

*Hardcoded health checks* — always run, not tied to any feature:

| ID | Name | Type | What it checks |
|---|---|---|---|
| `hc-daemon-alive` | Daemon heartbeat | db | `machines` table has a row for staging with `last_seen` < 2 minutes ago |
| `hc-dashboard-loads` | Dashboard renders | ui | Navigate to staging dashboard URL, assert no crash, assert pipeline view visible |
| `hc-ideas-page` | Ideas page loads | ui | Navigate to /ideas, assert tab bar visible |
| `hc-agent-inbound-poll` | Edge function responds | api | POST to `/functions/v1/agent-inbound-poll` with valid auth, assert 200 |
| `hc-realtime-connects` | Realtime connection | probe | Open a Supabase Realtime subscription, assert connected within 5s |
| `hc-submit-idea` | Idea submission | api | POST to create a test idea (tagged `qa-probe`), assert it appears in DB, clean up |

*Feature acceptance criteria* — read from `features.acceptance_criteria` for all features with status `merged` in the last 30 days:

For each feature, iterate its `acceptance_criteria` array and run each test case.

**Probe types**

`ui` — Use Playwright (headless Chromium). Navigate, interact, assert. The staging dashboard URL should be configurable via env var `STAGING_DASHBOARD_URL`.

`api` — HTTP request to a Supabase Edge Function endpoint. Assert status code and optionally response shape.

`db` — Query the staging Supabase DB via REST API. Assert row exists, count, or field value.

`probe` — Synthetic job injection. Insert a row into the `jobs` table with a minimal test payload tagged `qa-probe: true`. Poll for completion with a timeout. Clean up after regardless of outcome.

`code-review` — Call the Anthropic API with:
  - The feature description and acceptance criteria
  - The git diff from the feature's merge commit (fetch from GitHub API using the feature's `commit_sha` if available)
  - Prompt: "Given this feature spec and acceptance criteria, does this implementation correctly and completely deliver what was asked? Flag any gaps, mismatches, or concerns."
  - Parse the response for a pass/concern/fail judgment and a summary

**Result writing**

After each probe, write to `qa_results`:

```json
{
  "feature_id": "<uuid or null for health checks>",
  "test_case_id": "<tc-001 or hc-daemon-alive>",
  "status": "pass | fail | error | skip",
  "details": {
    "summary": "one-line human-readable result",
    "expected": "...",
    "actual": "...",
    "logs": ["..."],
    "duration_ms": 1234
  },
  "runner_version": "<git sha of workers binary>",
  "environment": "staging"
}
```

Write results incrementally as each probe completes — do not batch at the end. If the runner crashes mid-run, partial results are better than none.

**Failure response**

After writing a `fail` result, classify it:

*Auto-fixable* — attempt the fix, re-run the probe, update `auto_fixed = true` if it recovers:

| Failure | Fix action |
|---|---|
| `hc-daemon-alive` fails | SSH to staging machine and run `zazig-staging start` (if SSH access configured) OR post a Slack alert asking a human to restart |
| Credentials-related failure | Trigger credential refresh via Supabase edge function |

*Not auto-fixable* — create a bug idea in the zazig DB:

```json
POST /rest/v1/ideas
{
  "title": "QA failure: <test_case_id> — <summary>",
  "body": "<full details including steps, expected, actual, logs>",
  "status": "new",
  "type": "bug",
  "source": "qa-runner",
  "metadata": {
    "qa_result_id": "<uuid>",
    "feature_id": "<uuid or null>",
    "test_case_id": "<id>",
    "severity": "critical | high | medium | low"
  }
}
```

Set `qa_results.bug_idea_id` to the created idea's ID.

*Critical failures* — also send a Slack webhook notification immediately (don't wait for triage):
- Any `critical` severity failure
- More than 3 health check failures in one run
- `hc-daemon-alive` has been failing for 2+ consecutive runs

**Scheduling**

Run the full suite every 60 minutes. Configurable via env var `QA_INTERVAL_MINUTES`. Do not run concurrent suites — if a run is still in progress when the next one is due, skip the next one and log a warning.

**Architecture notes**

- This is a new worker type in Zazig Workers following the existing worker pattern in that repo
- Look at how Athena handles its agent workers for the structural pattern
- The runner itself should have no dependency on the zazig daemon being healthy
- Playwright can run headlessly — embed the Playwright binary or use the system installation
- Keep each probe isolated — a failure in one probe should never abort the rest of the run
- Probe timeouts: UI probes 30s, API probes 10s, probe type 60s, code-review 30s

## Phase 3: Failure Response

**Where:** zazig repo
**When:** after Zazig Workers QA Runner is writing to `qa_results`

The bug-filing logic lives in the Zazig Workers runner (Phase 2). What zazig needs is the receiving end:

**Ideas table** — ensure `type` supports `bug` and `source` supports `qa-runner`. These may already exist; confirm and add a migration if needed.

**Triage routing** — ideas with `source = 'qa-runner'` should surface visibly in the inbox. Consider a visual indicator in the ideas UI (a bug icon, different border colour) so Chris can distinguish QA-filed bugs from CPO-filed ideas.

**No edge function needed** — the runner writes directly to `ideas` via the service role key. No intermediary required.

## Phase 4: Promote Gate

**Where:** `packages/cli/src/commands/promote.ts`
**When:** after Phase 2 has been running for at least a week (so there are meaningful results to check)

Before the promote proceeds, query `qa_results` for the last 4 hours on staging:

```typescript
const cutoff = new Date(Date.now() - 4 * 60 * 60 * 1000).toISOString();
const failures = await supabase
  .from("qa_results")
  .select("test_case_id, details, feature_id, auto_fixed")
  .eq("environment", "staging")
  .eq("status", "fail")
  .eq("auto_fixed", false)
  .gte("run_at", cutoff);
```

**Soft block** (recommended): if failures exist, print a warning with detail and ask for confirmation. Don't hard-block — there may be legitimate reasons to promote with a known failing test. Log the override.

```
⚠️  QA checks failed in the last 4 hours on staging:
  - hc-daemon-alive: Daemon last seen 47 minutes ago
  - tc-002 (feature: Reject button): Idea status not updated after click

Proceed anyway? (y/N)
```

**Hard block** on `critical` severity failures with no auto-fix and no prior override.

## What This System Cannot Test

- **Agent output quality at scale** — the code-review probe gives a qualitative signal, but cannot catch every subtle implementation error. Human review of the actual feature output remains necessary for high-stakes changes.
- **Multi-machine coordination** — the runner tests one staging machine at a time.
- **Long-running reliability** — an hourly probe won't catch slow degradation (memory leaks, multi-day auth token decay). This requires longer observation windows.
- **Load behaviour** — single-probe runs don't simulate concurrent job execution.

These are acceptable gaps. The system covers ~85% of what Chris currently tests manually. The remaining 15% is inherently qualitative or requires production-scale observation.

## Implementation Order

1. Migration: `acceptance_criteria` column + `qa_results` table
2. Spec worker worker role + prompt
3. Zazig Workers: QA runner (external)
4. Ideas UI: bug source indicator
5. Promote gate

Steps 1–2 can be built now. Step 3 is the external brief above. Steps 4–5 follow once step 3 is running and producing results.
