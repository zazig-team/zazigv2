Org Model / Gateways

Gateways & Messaging

Gateways are communication channels between workers and the outside world. The agent never knows which platform it's talking to — it receives an opaque conversationId.

Gateway Types by Tier

Gateway access follows the three-tier model. Higher tiers get richer communication capabilities; lower tiers get none.

Executives

  • Access: Fully bidirectional
  • Model: Founders message CPO via Slack, get responses in-thread
  • Engagement: Primary interaction model — executives are always reachable
  • Platforms: Slack (primary), Discord (planned)

CPO, CTO, CMO — full conversational access

Employees

  • Access: Scoped, typically write-only
  • Model: Social Media Manager posts to X but doesn't receive DMs
  • Engagement: Per-role configuration — not all employees have gateways
  • Platforms: Varies by role function

Social Media Manager, DevOps — selective outbound

Contractors

  • Access: No gateways
  • Model: No direct engagement with external platforms
  • Engagement: Communicate only through job results and internal state
  • Platforms: None

All contractor roles — pure task execution

Architecture

Two distinct flows handle inbound and outbound messaging. The agent never interacts with platform APIs directly — everything goes through Edge Functions and the local MCP server.

Inbound Flow

External messages arrive at the agent via platform webhooks, an Edge Function adapter, and Supabase Realtime.

1. Platform
Slack / Discord
User sends a message mentioning or DMing the agent
2. Edge Function
slack-events
Verifies signature, deduplicates by event_id, generates conversationId
3. Routing Check
Agent Online?
Check heartbeat status of target agent's machine
4a. Online
Broadcast MessageInbound via Realtime
Local agent receives, injects into tmux session

If the agent is offline, the Edge Function replies with an "agent offline" message in the original thread. No queuing — the sender knows immediately.

Outbound Flow

Agents send messages through the local MCP server, which relays to an Edge Function for platform delivery.

1. Agent
Calls send_message(conversation_id, text)
MCP tool invocation within the agent's tmux session
2. MCP Server
agent-mcp-server (local stdio)
HTTP POST to Edge Function with conversationId and text
3. Edge Function
agent-message
Parses conversationId prefix to determine adapter, posts via platform API
4. Platform
Slack / Discord
Message appears in the original thread or channel

Wire Protocol

Messages between Edge Functions and local agents follow a typed protocol. The conversationId is opaque to the agent — only the adapter knows how to parse it.

MessageInbound 
interface MessageInbound {
  type: "message_inbound";
  protocolVersion: 1;
  conversationId: string;  // opaque
  from: string;
  text: string;
}
MessageOutbound 
interface MessageOutbound {
  type: "message_outbound";
  protocolVersion: 1;
  jobId: string;
  machineId: string;
  conversationId: string;  // opaque
  text: string;
}

conversationId Format

The conversationId encodes routing information as a prefixed string. The agent treats it as opaque — only the platform adapter parses it.

Format 
{adapter}:{routing-data}

// Examples:
slack:{team_id}:{channel}:{thread_ts}
discord:{guild_id}:{channel_id}:{message_id}

Why opaque? The agent calls send_message(conversationId, text) without knowing whether it's replying to Slack, Discord, or any future platform. This makes adding new adapters a server-side change only — no agent code changes required.

Platform Adapters

Each external platform gets an adapter stack: an inbound Edge Function, an outbound Edge Function, and optionally an OAuth flow.

slack-events

Inbound Edge Function. Handles url_verification challenge, verifies Slack request signatures, deduplicates by event_id, routes messages to the target agent via Realtime broadcast.

agent-message

Outbound Edge Function. Receives messages from agents, parses the conversationId prefix to determine which adapter to use, posts via the platform's API (e.g., Slack chat.postMessage).

slack-oauth

OAuth callback Edge Function. Handles the OAuth code exchange to obtain a bot token for the workspace. Stores tokens securely for outbound message posting.

agent-mcp-server

Local MCP server running via stdio on each machine. Exposes the send_message tool to agents. Relays outbound messages to the agent-message Edge Function via HTTP POST.

Open Decision

Dual Injection Path Conflict. The triggers design doc identified a conflict: this messaging system injects Slack messages directly into tmux, but the events queue subsystem also injects into tmux. Two separate paths writing to the same tmux session creates a race condition.

Recommended: Unify both paths through the events queue — Slack messages become events, events drive tmux injection. Single writer, no race.

Outstanding decision: When the agent is offline, should inbound messages bounce (immediate "offline" reply, message lost) or queue (store and deliver when agent comes back online)?
