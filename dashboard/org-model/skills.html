Skills Layer

Skills are procedural capability files — they teach agents how to work on specific types of tasks. Loaded per-job from ~/.claude/skills/.

What Skills Are

Skills are Layer 3 in the prompt stack. They are markdown files loaded at dispatch time by the executor. The executor reads skill file content and prepends it to context after the role prompt and before task context.

If a skill file is missing, the job continues with a warning only — skills are non-blocking. This keeps the system resilient: a misconfigured skill path never tanks a job.

Per-Role Skill Sets

Each role has a defined set of skills configured in the roles table. Skills are loaded automatically when a job is dispatched to that role.

Role Skills
CPO standup, cardify, review-plan, cpo, scrum, brainstorming
CTO cto, multi-agent-review
Senior Engineer commit-commands:commit
Reviewer multi-agent-review
Junior Engineer (none — runs on Codex)
Researcher (none — tool-driven)
Product Manager deep-research, second-opinion, repo-recon, review-plan, brainstorming, cardify

The commit-commands:commit syntax means the commit-commands skill file with the commit sub-command. Skills can have colon-delimited variants.

Skills vs Doctrines vs Canons

Three systems that teach agents different things. They complement each other — skills teach procedure, doctrines teach judgment, canons teach domain knowledge.

Skills

  • Teaches: How to work
  • Nature: Procedural capability
  • Scope: Per-role
  • Loading: Every job at dispatch
  • Format: Markdown files on disk
  • Location: ~/.claude/skills/
  • Example: "Here's how to run a standup"

Doctrines

  • Teaches: What they believe
  • Nature: Professional judgment
  • Scope: Per-role
  • Loading: Proactively injected
  • Format: Structured claims in DB
  • Location: Supabase doctrines table
  • Example: "Tests should be written before code"

Canons

  • Teaches: What they've studied
  • Nature: Reference knowledge
  • Scope: Company-wide
  • Loading: Mostly reactive retrieval
  • Format: Massive document collections
  • Location: Supabase + vector store
  • Example: "The OWASP Top 10 says..."

Skill Sync

Skills are distributed to local machines via a launchd-based sync daemon. The repo is the single source of truth — local machines pull and symlink.

Architecture

  • Source of truth: .claude/skills/ in the zazigv2 repo
  • Sync interval: git pull --ff-only every 15 minutes via launchd
  • Local path: Symlinks from ~/.claude/skills/ point to repo checkout
  • Direction: One-way — repo to local, never the reverse

Design Decisions

  • Symlinks, not copies: git pull updates source files; symlinks automatically resolve to latest
  • Personal skill protection: Real directories (not symlinks) are never overwritten by sync
  • Graceful failure: If pull fails (network, conflicts), existing skills remain in place
  • Onboarding: One command — ./scripts/install-skill-sync.sh

Sync Flow

Source
zazigv2 repo .claude/skills/
Skill files checked into version control
Sync
git pull --ff-only
launchd runs every 15 minutes on each machine
Local
Symlink to ~/.claude/skills/
Agent reads skills from here at dispatch time

Why symlinks? A copy-based approach would require re-copying on every sync. Symlinks mean the moment git pull lands new content, every agent on that machine sees it immediately — zero propagation delay after pull.
