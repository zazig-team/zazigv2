The Problem

Four proposals designed the same system independently.

📋

Two Table Schemas

Ideas Inbox and Ideaify both defined the ideas table with incompatible field names, different scopes, and conflicting design philosophies.

🔧

Duplicate MCP Tools

Both proposals defined create_idea and query_ideas with different parameter signatures. No single source of truth.

🔀

Blurred Boundaries

Processing and triage responsibilities overlapped. The Telegram bot did its own processing instead of routing to Ideaify.

Layer 1

Capture

Receive raw input from any channel. Transcribe voice. Route to processing. Nothing more.

Capture channels are thin adapters. They receive a message, optionally transcribe voice to text, and hand off to Ideaify. They do not clean, categorise, or store ideas.

This means every channel benefits from the same processing quality. Ideaify improves once, everywhere. Adding a new channel (email, WhatsApp, web form) is trivial — just route text.

Voice transcription belongs here because it transforms a non-text format into text. It's mechanical, not intelligent. Recommended: gpt-4o-transcribe.

Capture Channels

  • 💻Terminal — CPO receives, runs ideaify
  • ✈Telegram — Voice + text, transcribe, hand off
  • 💬Slack — Text, route to ideaify
  • 🤖Agents — Already structured, skip ideaify
  • 🌐Web / API — Direct POST to edge function
Layer 2

Process (Ideaify)

Clean, split, and categorise raw input into structured idea records. Data cleaning, not product judgment.

Ideaify's most valuable function is multi-idea splitting. A 10-minute voice note covering 5 topics becomes 5 clean, individual idea records. Conservative splitting — when in doubt, don't split.

Each idea gets categorised by scope (job/feature/initiative), complexity, domain (product/engineering/marketing), and autonomy level. Ambiguity is flagged, not resolved.

Phase 1: CPO runs ideaify as a skill. Phase 2: Ideaify becomes an intake-processor contractor, dispatched automatically. Same skill, different runner.

Ideaify does

  • Clean raw text into clear prose
  • Split multi-idea inputs
  • Categorise by scope and domain
  • Tag with metadata
  • Flag ambiguity and duplicates

Ideaify does not

  • Judge whether ideas are good
  • Research feasibility
  • Prioritise ideas
  • Create features or jobs
  • Ask for clarification
Layer 3

Store (Ideas Inbox)

One Supabase table. One schema. Five MCP tools. The single source of truth.

The canonical ideas table merges the best of both archived schemas. raw_text preserves the original input verbatim. title + description hold the cleaned version from Ideaify.

source + originator are separate fields. Source = channel (where it came from). Originator = actor (who said it). "CTO noticed a problem" = source: terminal, originator: cto.

Full traceability: idea_created → idea_triaged → idea_promoted → feature_created. Every feature traces back to the original raw thought.

MCP Tools

  • +create_idea — insert a new idea
  • ?query_ideas — search + filter
  • ✎update_idea — triage fields
  • ↑promote_idea — atomic promote
  • ++batch_create_ideas — multi-idea splits
Layer 4

Triage (CPO Sweep)

The CPO reviews, enriches, prioritises, and decides: promote, park, or reject.

Triage is a behavioural addition to the CPO's existing role, not a separate system. During standup or conversation start, the CPO sweeps the inbox for new ideas.

For each idea: review, refine if needed, set priority, recommend an action. The CPO prepares ideas for human decision.

Trust boundary evolves over time. v1: always ask. v2: CPO can promote small items. v3: full autonomy. Trust is earned through demonstrated good triage.

Idea Lifecycle

new → triaged → promoted
or → parked or rejected

Parked ideas can resurface. Rejected ideas are preserved for context.

Layer 5

Present (Idea Visualiser)

Render ideas and documents for human review at every lifecycle stage. Approve from your phone.

The visualiser works at multiple stages: raw ideas as compact cards, triaged ideas with enrichment, proposals as full documents, features as pipeline status trackers.

Each view includes interactive actions: approve, reject, comment, request changes. Deep-links back to the CPO conversation close the loop between mobile review and terminal action.

Hosted as a private Netlify deployment. Vanilla HTML + Mermaid.js. Reads from Supabase and the docs/plans/ repo directory.

Renders at Every Stage

  • 📝Raw idea — compact card with flags
  • ⚖Triaged idea — enriched with recommendation
  • 📄Proposal — full doc with decisions
  • 🏗Feature — pipeline status tracker

How Ideas Flow

Three real scenarios showing the full pipeline in action.

🎤 Voice Note on the Go

Tom records a rambling voice note while walking the dog. It contains 3 ideas.

Telegram receives → Transcribe voice → Ideaify splits 3 ideas → 3 records in inbox → CPO sweeps + triages → Tom reviews on phone → Approves promotion

💻 Terminal Conversation

"We should fix the auth timeout and also add rate limiting."

CPO receives text → Ideaify splits 2 ideas → 2 records in inbox → CPO triages immediately → Auth → job, Rate limit → feature

🤖 Agent-Discovered Opportunity

Monitoring Agent spots a competitor feature worth investigating.

Agent creates idea → Structured, skips ideaify → CPO triages in sweep → Promotes to research → Proposal on Visualiser
