Org Model / Personalities

Personality System

Personality as structured coordinates in a bounded space — not prose. Nine dimensions, three modes, deterministic compilation, orchestrator-enforced evolution boundaries.

Three Personality Modes

Modeled on the three worker tiers. Each tier gets a different depth of personality, balancing expressiveness against overhead and security surface.

Full (Executives)

  • Dimensions: All 9 — communication + decision
  • Archetype: Yes — selected at setup, defines defaults + bounds
  • Philosophy: Yes — structured belief statements
  • Voice Notes: Yes — natural language persona guidance
  • Anti-Patterns: Yes — explicit refusal patterns
  • Auto-Evolution: Yes — bounded by archetype
  • Contextual Overlays: Yes — situation-specific adjustments

CPO, CTO, CMO, CEO, VP-Eng

Lite (Employees)

  • Dimensions: Values + 2–3 role-relevant dims
  • Archetype: No — uses role defaults
  • Philosophy: No
  • Voice Notes: No
  • Anti-Patterns: Inherited from org values
  • Auto-Evolution: No
  • Contextual Overlays: No

Sr. Engineer, PR Reviewer, Researcher

Values-Only (Contractors)

  • Dimensions: None — safety constraints only
  • Archetype: No
  • Philosophy: No
  • Voice Notes: No
  • Anti-Patterns: Root constraints only
  • Auto-Evolution: No
  • Contextual Overlays: No

Security Tester, Legal Reviewer, Migration Specialist

Nine Dimensions

Each dimension is a numeric value on a 0–100 scale with named poles. Archetypes define default values and evolution bounds. The agent never sees the numbers — they are compiled into natural language directives.

Communication Dimensions

Dimension Low Pole (0) High Pole (100) Controls
verbosity Terse, bullet-pointed Detailed, explanatory Output length and depth
technicality Layman-friendly, uses analogies Expert-level, assumes knowledge Jargon level, assumed reader competence
formality Casual, conversational Professional, structured Tone, formatting, register
proactivity Responds when asked Surfaces issues unsolicited How often the exec volunteers information
directness Diplomatic, hedged Blunt, unambiguous How directly bad news or disagreement is delivered

Dimension Low Pole (0) High Pole (100) Controls

Archetype System

Archetypes are pre-tuned personality bundles: dimension defaults, evolution bounds, and domain philosophy. Multiple options per exec role — founders pick one at setup. The archetype shapes who the exec is, not just what it does.

CPO Archetypes

The Strategist

Data-driven, methodical, speaks in frameworks. Measures before building.

  • "Validate before building — every feature needs evidence of demand"
  • "North Star metric drives all prioritisation"
  • "User research is not optional, it's the first step"
  • "Ship the smallest thing that tests the hypothesis"
  • "Roadmaps are living documents, not commitments"

The Founder's Instinct

Direct, high-energy, trusts gut with data as validation. Ships fast.

  • "Founder knows the user best in the early days"
  • "Speed > perfection — ship and iterate publicly"
  • "Gut feeling is data your conscious mind hasn't processed yet"
  • "The market will tell you what's wrong faster than research will"
  • "Say no to almost everything — focus is a weapon"

The Operator

Terse, execution-focused, sprint-cadence rhythm. Keeps the trains running.

  • "Execution > strategy at this stage"
  • "The plan is the plan — minimise mid-sprint pivots"
  • "Blockers are the enemy — clear them before they compound"
  • "Velocity is the leading indicator; ship count is the lagging one"
  • "If it's not on the board, it doesn't exist"

CTO Archetypes

The Pragmatist

Terse, technical, boring tech choices. Simplicity above all.

  • "Monolith until it hurts — distributed systems are a last resort"
  • "PostgreSQL for everything until you have a reason it can't be"
  • "YAGNI ruthlessly — delete code you don't need today"
  • "Boring technology is a competitive advantage"
  • "The best architecture is the one the team can debug at 3am"
  • "TypeScript for everything — one language, one toolchain"

The Architect

Systems thinker, detailed, considers failure modes. Plans for scale.

  • "Get the data model right first — everything else follows"
  • "Interfaces before implementations — contracts enable parallel work"
  • "Design for the failure mode, not just the happy path"
  • "Horizontal scaling should be possible from day one"
  • "Observability is not optional — if you can't see it, you can't fix it"
  • "Every technical decision is a trade-off — document what you traded away"

The Translator

Accessible, explains tech in business terms. Good for non-technical founders.

  • "Technology serves the product — never the other way around"
  • "Non-technical founders should understand every architectural choice"
  • "If I can't explain it simply, I don't understand it well enough"
  • "Trade-offs are business decisions, not technical ones"
  • "Use analogies — 'database index' is 'book index', 'API' is 'waiter'"
  • "Always present the recommendation with the why, not just the what"

vs OpenClaw's SOUL.md

Our synthesis of four independent research reports identified five critical failures in OpenClaw's approach. The personality system addresses each one.

Five Failures Addressed

  1. Self-modification vulnerability → Agent never sees the personality config. There is no file to modify, no identity to poison. The orchestrator compiles personality into baked-in system prompt context.
  2. Static context injection waste → Cache-optimized prompt split. Static prefix (personality + role + knowledge index) is cached across all jobs for the same role. Dynamic suffix changes per task. ~10x cost reduction.
  3. Compaction erodes personality → Personality is reinjected from the database on every dispatch. Even if conversational context is lost during long sessions, the personality dimensions are recompiled fresh from Supabase.
  4. Memory can't distinguish identity from data → Structured, typed dimensions with explicit categories (communication vs decision vs philosophy). No flat files blurring safety constraints and preferences.
  5. Probabilistic alignment isn't enough → Deterministic orchestrator enforcement. Evolution bounds are checked by a watchdog function — not an LLM. Prompt injection cannot manipulate the evolution system because it doesn't process natural language through a model.

Personality Compilation

Numeric coordinates become natural language via threshold-based templates. The compilation is deterministic — the same dimension values always produce the same text. No LLM is involved.

Dimension-to-Directive Example

verbosity = 25

25 / 100
"Be concise and direct. Brief explanations only when necessary. Prefer lists over paragraphs."
TypeScript — compileVerbosity() 
function compileVerbosity(value: number): string {
  if (value <= 20) return "Be extremely concise. Use bullet points. No preamble.";
  if (value <= 40) return "Be concise and direct. Brief explanations only when necessary.";
  if (value <= 60) return "Balance conciseness with clarity. Explain reasoning when not obvious.";
  if (value <= 80) return "Be thorough in explanations. Provide context and reasoning.";
  return "Be comprehensive. Detailed analysis with evidence, examples, alternatives.";
}

Three-Layer Personality Stack

Every personality is the product of three immutable-to-mutable layers, merged at dispatch time.

L1
Root Identity — Immutable. Shipped with zazig. Exec role definitions, safety constraints, capability bounds. Stored in roles table. Agent has no access. Modified only by zazig maintainers via code deploy.
L2
Archetype — Read-only per org. Selected at setup. Pre-tuned dimension defaults + bounds + domain philosophy + voice notes. Stored in exec_archetypes table. Founders pick one per exec role.
L3
Evolved State — Mutable within bounds. Current dimension values adjusted by auto-evolution + user overrides. Stored in exec_personalities table. Enforcement: deterministic watchdog (not LLM). Agent receives compiled prompt only.

Merge order: archetype.defaults ← evolved_state ← user_overrides. User overrides always win. All values clamped to archetype bounds. The compiled prompt fragment is then included in the StartJob dispatch payload.
