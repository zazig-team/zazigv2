The missing layer: what causes work to happen and how subsystems communicate about it. Seven subsystems + three cross-cutting capabilities.

1. Heartbeat System

Per-job health reporting inside the existing heartbeat envelope. The local agent inspects each running agent session and reports structured health data every 30 seconds.

Orchestrator Decision Table

Condition Action
stuck for 2 consecutive heartbeats Log warning, increment stuck_count
stuck_count >= threshold (default 3 = 90s) Kill + requeue
permissionBlocked Notify human via Slack
tmuxSessionAlive false Requeue as failure
CPO context > 200 turns + compacted Auto-restart CPO
No heartbeat for 2 minutes Mark machine dead, requeue all jobs

Escalation chain: stuck → warning → kill → requeue. Each step is logged. Permission blocks go directly to a human because only humans can approve tool use.

2. Scheduler / Cron

Postgres function run_scheduler() called by pg_cron every minute. Two modes of operation depending on whether the scheduled task should run independently or relay through an existing agent.

Isolated Mode

Creates a standard job in the jobs table. Dispatched normally through the orchestrator to an available slot.

Relay Mode

Enqueues a system event targeting wake_target (e.g., CPO). The target agent picks up the event via the wake service.

Seed Jobs

Name Schedule Mode Purpose

nightly-bug-scan 0 4 * * * Isolated Scan for antipatterns
standup-reminder 0 9 * * 1-5 Relay (CPO) Initiate standup
pipeline-health */30 * * * * Relay (CPO) Check pipeline state
roadmap-review 0 10 * * 1 Relay (CPO) Weekly roadmap check

3. Wake Service

The poke mechanism that wakes agents when they have pending events. Events are coalesced to prevent interrupt storms — multiple events arriving within a window are batched into a single wake.

Step 1
Events Enqueue
Cron, hooks, or external source writes to agent_events
Step 2
Wake Poke
Realtime subscription fires on the target machine
Step 3
Coalesce
Batch pending events within a window to avoid interrupt storms
Step 4
Cheap-Check
Lightweight triage: is this event still relevant? Is the agent idle?
Step 5
Inject
Coalesced events injected into the agent session
Step 6
Ack
Agent acknowledges processing, events marked consumed

4. Events Queue

The agent_events table in Postgres. All events are trust-classified at creation time. Trust classification determines how the event is validated and injected.

Internal Events

Trust level: internal. Produced by cron, hooks engine, and orchestrator state transitions. No external validation required.

  • Scheduled wakes
  • Pipeline state changes
  • Hook-triggered actions
  • Job lifecycle events

External Events

Trust level: external. Produced by webhooks, GitHub, CI callbacks. Verified via HMAC signatures before processing.

  • GitHub PR merged
  • CI pipeline passed/failed
  • Webhook callbacks
  • Third-party integrations

All injection goes through the coalesced pipeline regardless of trust level. The difference is in validation — external events must pass HMAC verification and rate limiting before entering the queue.

5. Hooks Engine

Policy enforcement layer. Composable hooks run before and after job state transitions. Each hook is a pure function: inspect state, return allow/deny + optional side effects.

Budget Check

Verifies the job's estimated cost fits within the team's remaining budget for the billing period. Blocks dispatch if budget would be exceeded.

Safety Gates

Pre-dispatch validation: does the agent have the required permissions? Is the target branch protected? Are credentials available?

Slot Validation

Confirms the target machine has an available slot matching the job's model requirement (Claude Code vs Codex).

Hooks are composable — multiple hooks chain together in a pipeline. A job must pass all pre-hooks before dispatch. Post-hooks handle cleanup, logging, and event emission.

6. Lanes

Two-lane execution model prevents system maintenance from blocking user-initiated work. Each lane has its own queue and slot allocation.

Main Lane

  • User-dispatched jobs (from cards)
  • Scheduled isolated jobs
  • CPO-initiated work

Priority: High. Gets first claim on available slots.

Background Lane

  • Verification runs
  • Code review jobs
  • System maintenance
  • Nightly archivers and scans

Priority: Low. Only uses slots not claimed by Main lane.

If all slots are occupied by Main lane work, Background lane jobs wait. Background work never starves user-facing work.

7. External Triggers

Events originating from outside the system: GitHub webhooks, CI pipeline callbacks, third-party integrations. All external triggers carry trust level external and require cryptographic verification.

8. Emergency Stop (ESTOP)

Kill broadcast across all machines. When activated, every running agent is halted and every queued job is frozen. Used when agents go rogue or burn budget.

ESTOP Activated
Broadcast kill to all machines
Immediate
Freeze all jobs
Running → frozen, queued → frozen
Immediate
Kill all agent sessions
Local daemons terminate all tmux sessions
Manual
Founder reviews and lifts
ESTOP remains active until manually cleared

ESTOP triggers: Runaway budget consumption, agent writing to production systems, detected prompt injection, agent-to-agent feedback loops, or any founder-initiated kill switch.

9. Restart Recovery

After an ESTOP or daemon restart, the system must recover without duplicating work or allowing stale agents to corrupt requeued jobs. The key mechanism: dispatch attempt fencing.

Dispatch Fencing 
// Each dispatch generates a unique attempt ID
dispatch_attempt_id: UUID

// The agent receives this ID at launch
// Every state mutation checks: does my attempt ID match the job's current attempt?
// If not → this agent is stale → refuse to write, exit gracefully

// On requeue:
// 1. New dispatch_attempt_id generated
// 2. Old agent's writes rejected by fencing check
// 3. New agent picks up clean state
Step 1
ESTOP / Restart
All agents terminated, jobs frozen
Step 2
Requeue Frozen Jobs
Each requeued job gets a new dispatch_attempt_id
Step 3
Fence Check
Stale agents with old attempt IDs cannot write state
Step 4
Clean Dispatch
New agents pick up jobs with guaranteed clean state

Dispatch attempt fencing is the mechanism that makes the entire trigger system crash-safe. Without it, a restart could leave zombie agents writing stale results to requeued jobs.
