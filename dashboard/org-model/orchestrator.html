Orchestrator & Dispatch

Deterministic cloud orchestration — no LLM in the control plane. Supabase state, local execution, slot-based resource management.

Three-Layer Architecture

The orchestrator is a deterministic server in the cloud. Local machines run thin agent daemons that receive commands via websockets and execute them locally. CPO is the sole persistent agent.

When the orchestrator dispatches a job, it reads the worker's tier and role, then assembles the prompt stack through an 8-step process.

From roles.prompt
Step 4

From roles.skills[]

Cache-optimized order: static prefix (personality + role + knowledge index) then dynamic suffix (per-task content)

Sent to local agent via Supabase Realtime websocket

Slot System

Concurrency is constrained by the team's combined API plan limits. Card complexity determines which slot type the orchestrator assigns. The orchestrator prefers the machine with the most available slots (load balancing).

Complexity Slot Type Model Tier Use Case
simple Codex Codex (cheapest) Mechanical changes, boilerplate, straightforward fixes
medium Claude Code Sonnet Multi-file changes, moderate reasoning required
complex Claude Code Opus Architecture decisions, deep investigation, multi-step reasoning

Load balancing: If all slots of the needed type are occupied, the card queues until a slot frees. The orchestrator always prefers the machine with the most available slots of the required type.

Review Chain Pattern

The core cost optimisation: cheap models produce, expensive models review. This is the produce-review pattern — get 90% of the value at 10% of the cost, then spend the remaining budget on targeted review.

Junior Engineer

Produce: Qwen 3.5 (free, local)

Review: Codex (~$0.01)

~$0.01 per task

vs Sonnet for everything: ~$0.10

Senior Engineer

Produce: Codex (~$0.01)

Review: Sonnet (~$0.05)

~$0.06 per task

vs Opus for everything: ~$0.50

The pattern is recursive: if the reviewer rejects, feedback is attached to the card and the orchestrator re-dispatches to the producer. The cheap model learns from the expensive model's feedback within the session.

CPO Failover

CPO is the sole persistent agent and the human interface. Its availability is the critical path. The orchestrator monitors CPO health via the host machine's local agent heartbeats (30s intervals).

Under 15 Minutes

Orchestrator waits. Handles brief sleep, restarts, network blips. No migration triggered — most outages resolve within this window.

Over 15 Minutes

Orchestrator migrates CPO to another available machine. The new host gets hosts_cpo: true, the old one gets it cleared.

No ping-pong: When the original machine comes back online, CPO stays on the new host. This prevents repeated migrations if the original machine is unstable. The orchestrator assigns CPO to the new host permanently until the next failover event.
