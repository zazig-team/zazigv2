## 2. Hash-Anchored Edits (Hashline System)

This is the most interesting technical contribution. It replaces the standard "search for string, replace with new string" edit pattern used by Claude Code, Cursor, etc.

### How It Works

1. **Anchor computation:** Every line gets a 2-character hash anchor computed via xxHash32 on whitespace-stripped content. Lines with no alphanumeric characters (pure punctuation/braces) use the line index as seed to reduce collisions.

2. **Custom encoding:** The 8 lowest bits of xxHash32 output map to a 256-entry dictionary using alphabet `ZPMQVRWSNKTXJBYH` (4-bit nibble encoding, 2 chars per hash).

3. **Presentation format:** When the model reads a file, each line appears as:
   ```
   LINENUM#HASH:CONTENT
   ```
   Example: `5#VR:function foo() {`

4. **Edit operations:** The model specifies edits using anchors, not by reproducing text:
   ```json
   { "op": "replace", "pos": {"line": 5, "hash": "VR"}, "end": {"line": 8, "hash": "QM"}, "lines": ["new content"] }
   { "op": "append", "pos": {"line": 5, "hash": "VR"}, "lines": ["inserted after line 5"] }
   { "op": "prepend", "pos": {"line": 5, "hash": "VR"}, "lines": ["inserted before line 5"] }
   ```

5. **Staleness detection:** Hash mismatches between the anchor the model references and the current file state are caught before mutation. If the file changed since the model last read it, the edit is rejected rather than applied to wrong content.

### Why It Matters

Standard string-matching edits (used by Claude Code, Cursor, Aider) fail when:
- The model reproduces whitespace incorrectly
- Indentation doesn't match exactly
- The target string appears multiple times
- The model hallucinates surrounding context

Hash anchors eliminate all of these by never requiring the model to reproduce existing text. The model only needs to reference a line number + 2-char hash.

### Benchmark Results

Tested across 16 models on React component editing tasks. Results were dramatic:
- Some models improved from **6.7% to 68.3% success rate** — roughly 10x improvement
- The improvement is most pronounced on weaker/smaller models where string reproduction is less reliable
- Even strong models showed measurable gains

### Application Pipeline

The patch applicator supports both modes:
- **Hashline mode:** Anchor-based, described above
- **Standard diff mode:** Falls back to fuzzy sequence matching with three-tier search (content-based, line-hint, full-file scan) and progressive relaxation thresholds

## 3. LSP Integration

### Architecture

- **40+ languages supported:** Rust, Go, Python, TypeScript, Java, Kotlin, and more
- **Auto-discovery:** Finds project-local servers in `node_modules/.bin` and `.venv/bin`
- **Multiplexer (lspmux):** Shares expensive servers (especially rust-analyzer) across multiple sessions, reducing startup overhead and memory footprint
- **Format-on-write:** After the edit tool writes a file, the LSP server applies formatting automatically
- **Real-time diagnostics:** Diagnostics arrive via `textDocument/publishDiagnostics` and are stored per-URI with version tracking
- **Workspace edits:** Server-initiated `workspace/applyEdit` requests flow back through the client

### Integration With Edit Pipeline

The LSP client sits downstream of the edit tool:
1. Model specifies edit (hashline or diff)
2. Edit is applied to disk
3. LSP client syncs content (`syncContent()`)
4. Formatting applied if server supports it
5. Diagnostics collected and fed back to model
6. File locks prevent concurrent modifications on same URI

### lspmux Detail

Transparent multiplexer for expensive language servers. Currently targets rust-analyzer specifically (high startup cost, high memory). Wraps commands as `lspmux client -- [original args]` with env var pointing to actual server binary. Falls back to direct spawn if unavailable. Detection results cached for 5 minutes.

## 4. Subagent System (Task Tool)

### Bundled Agents

Six agents ship by default:
- **explore** — codebase navigation and understanding
- **plan** — multi-step task planning
- **designer** — UI/UX design tasks
- **reviewer** — code review with structured findings
- **task** — general-purpose subagent
- **quick_task** — lightweight variant

### Discovery and Override

Agents discovered from multiple sources in priority order:
1. `.omp/agents/`, `.claude/agents/`, `.codex/agents/`, `.gemini/agents/` (project-level)
2. Same paths at user level (`~/.omp/agents/`, etc.)
3. Plugin roots with `agents/` subdirectories
4. Bundled agents (lowest priority)

First-wins deduplication by name — project agents override user agents override bundled.

### Parallel Execution

Worker pool pattern with configurable concurrency limit. Workers share a queue via atomic index counter. Results maintain input order. Error handling is fail-fast: first non-abort error cancels remaining work. Abort signals produce partial results with `aborted: true` flag.

### Git Worktree Isolation

Each subagent gets its own git worktree for filesystem isolation:
1. `ensureWorktree()` creates a detached-HEAD worktree at current commit
2. `captureBaseline()` records HEAD hash, staged changes, unstaged changes, untracked files
3. `applyBaseline()` restores the full working state in the worktree
4. Subagent executes in isolation — can modify files without affecting parent or siblings
5. `captureDeltaPatch()` extracts the diff of changes made by the subagent
6. `cleanupWorktree()` removes worktree via `git worktree remove -f`

Nested git repos (non-submodule) are copied wholesale with baseline commits for clean delta tracking.

### Artifact Streaming

Real-time progress via event-driven architecture:
- Progress events coalesce at 150ms intervals
- Tracks: last 8KB of output, active tool execution with argument previews, token accumulation, last 5 tool invocations
- Artifacts stream to disk: JSONL session files, output markdown, line/character count metadata
- `submit_result` provides authoritative output; fallback parses JSON from raw output against schema

### Execution Lifecycle

1. Validate abort signals, resolve model overrides
2. Configure tools (selective inclusion from agent definition), system prompts, extensions
3. Execute initial task prompt
4. Retry up to 3 times with reminder prompts if `submit_result` not called
5. Finalize: explicit result > JSON extraction > finding aggregation
6. MCP connections proxied from parent (no redundant server spawns)

### Recursion Control

Task tool auto-included unless max recursion depth reached. Spawn policies validated at execution time. Agents can be discoverable but unavailable due to guardrails.

## 6. Optimized Tool Harness

The "optimized tool harness" refers to the combined effect of several performance and reliability systems:

### Rust N-API Native Engine (~7,500 lines)

Platform-specific compiled modules for performance-critical operations:

| Module | Purpose |
|--------|---------|
| grep | Regex search with glob filtering (ripgrep internals) |
| shell | Embedded bash with persistent sessions (brush-shell) |
| text | ANSI-aware width, truncation, wrapping |
| keys | Kitty keyboard protocol parser |
| highlight | Syntax highlighting (11 color categories, syntect) |
| glob | Filesystem discovery with gitignore respect |
| task | Async work scheduler |
| ps | Cross-platform process management |
| prof | Circular-buffer profiler |
| image | PNG/JPEG/WebP/GIF encode/decode |
| clipboard | System clipboard access |
| html | HTML-to-Markdown conversion |

Supports x64 (with AVX2 "modern" variant) and arm64 on Linux, macOS, and Windows.

### Filesystem Caching

Shared `fs_cache` maintains normalized directory entries keyed by canonical root + hidden-file inclusion + gitignore status. "Stale-empty recheck" policy triggers re-scan when cached queries return zero results past age threshold. Sequential processing when offset/maxCount limits set (preserves deterministic ordering), parallel otherwise.

### Tool Registration Architecture

Extension-based tool registration with:
- `AbortSignal` for cancellation
- `onUpdate()` for streaming intermediate results
- Middleware-style `tool_call`/`tool_result` interception (runs in extension order, each sees prior modifications)
- `onSession()` lifecycle callbacks (start, switch, branch, tree operations)
- Session state persistence via `appendEntry()` with branch history reconstruction

### Bash Interception

Commands are intercepted before shell execution. Dedicated tools (grep, read, write) are preferred over raw bash equivalents, preventing the model from using shell commands when a purpose-built tool exists.

## 7. Additional Notable Systems

### TTSR (Time Traveling Streamed Rules)

Mid-generation correction without restarting:
1. Rules with `ttsrTrigger` regex patterns monitor assistant output during streaming
2. When a pattern matches, `agent.abort()` fires immediately
3. After 50ms delay, system decides: discard partial output or keep it
4. Synthetic `<system-interrupt>` message injected with corrective rule content
5. Generation resumes via `agent.continue()`
6. Repeat suppression: "once" (single trigger) or "after-gap" (re-trigger after N clean turns)

### Model Roles

Different models for different tasks: default, smol (fast exploration), slow (deep reasoning), plan, commit. Selected via interactive `/model` interface.

### Universal Config Discovery

Reads configuration from 8 AI tools: Claude Code, Cursor, Windsurf, Gemini, Codex, Cline, GitHub Copilot, VS Code. Discovers MCP servers, rules, skills, hooks, tools, slash commands, prompts, context files. First-wins deduplication by name across all sources.

### Session Management

JSONL-based with tree branching, manual compaction (`/compact`), automatic overflow recovery, and configurable context window. Smart boundary decisions never cut mid-tool-result. Pre-compaction prunes verbose tool outputs (protects newest 40k tokens, requires 20k minimum savings).

## Relevance to Zazig

### Direct Steal-List

1. **Hash-anchored edits** — The 10x improvement in edit reliability across weaker models is significant. If we ever build an in-house editing tool or need to improve edit success rates for agents, this approach is proven. The xxHash32 + 2-char nibble encoding is elegant and cheap to compute.

2. **TTSR (mid-stream correction)** — Correcting model output during generation without restarting the conversation is a pattern we could apply to agent guardrails. Instead of post-hoc rejection, inject corrections mid-stream. Zero context overhead when rules don't trigger.

3. **Worktree isolation for parallel agents** — Their subagent system uses git worktrees for filesystem isolation, which is the same pattern we use. Their baseline capture/restore and delta extraction are well-implemented reference implementations.

4. **LSP multiplexing** — Sharing expensive language servers across multiple agent sessions. Relevant if we ever run multiple code agents against the same repo simultaneously.

5. **Stealth browser automation** — 14 stealth scripts for bot detection evasion. Relevant for any monitoring or verification agents that need to interact with web services.

6. **Extension middleware for tool interception** — Their `tool_call`/`tool_result` middleware pattern enables policy enforcement across all tools. Could inform how we structure tool governance in the pipeline.

### Architecture Observations

- **Bun-native throughout** — Full migration from Node.js. Performance and startup gains.
- **Rust for hot paths** — N-API for grep, shell, text processing, image handling. ~7,500 lines. Keeps JS for orchestration, Rust for computation.
- **Config discovery across ecosystems** — Reads from Claude Code, Cursor, Windsurf, etc. Pragmatic interop rather than walled garden.
- **Capability-based discovery** — `defineCapability`/`registerProvider` pattern for modular features. Clean separation of interface from implementation.

### Not Directly Relevant

- Their commit tool (we don't need AI-powered commits in the pipeline)
- Python kernel integration (our agents don't run notebooks)
- SSH tool (our agents run locally on the machine)
- Theme system (we're headless)
