# Recon: tobi/qmd
*Analyzed: 2026-02-19 | Commit: 6ac7c68 | Compared against: zazigv2*

## TL;DR

- qmd is Tobi Lütke's on-device hybrid search engine: BM25 + vector + LLM reranking over markdown files, all local via SQLite + GGUF models
- zazig currently shells out to `qmd query --json` per search — cold models, 30s timeout, subprocess overhead
- Biggest win: switch to `qmd mcp --http --daemon` and replace subprocess with persistent `fetch`/httpx calls
- Context tree (hierarchical path metadata) is the most underrated feature — ranks results better with near-zero implementation cost
- Strong-signal gating and chunk-first rerank are underrated efficiency patterns worth understanding before adding memory search to zazigv2

## Steal List

Ranked by impact for zazigv2.

**1. HTTP daemon integration** — Stop shelling out. Run `qmd mcp --http --daemon` once at startup. Hit `POST localhost:8181/query` with JSON. Models stay warm in VRAM across calls. The `/health` endpoint lets you check liveness. Built-in daemon workflow already exists — PID file, log, clean shutdown. This is the single highest-leverage change: eliminates cold-start latency and the 30s timeout risk.

```python
# Before: cold subprocess per search
result = subprocess.run(["qmd", "query", "--json", "--", query], ...)

# After: persistent daemon + httpx
resp = await client.post("http://localhost:8181/query", json={
    "searches": [{"type": "lex", "query": query}, {"type": "vec", "query": query}],
    "limit": 5,
})
```

**2. Context tree** — qmd supports hierarchical path-prefix metadata: `qmd context add qmd://agents/cpo "CPO role memory"`. When search returns a result from that path, the context string is attached to the result. Agents get richer results without changing their queries. Cost: run `qmd context add` once per agent memory directory. This is what Codex flagged as most underrated — "quality per effort is excellent."

**3. Structured multi-query (lex + vec + hyde)** — Currently zazig sends a single query string. The `/query` endpoint accepts typed sub-queries with the first getting 2× weight. `lex` = BM25 keywords, `vec` = semantic, `hyde` = hypothetical document (write what the answer looks like). For agent memory retrieval, combining `lex` + `vec` outperforms either alone:

```json
{
  "searches": [
    {"type": "lex", "query": "sprint planning cpo"},
    {"type": "vec", "query": "what was decided about the roadmap priority last quarter"}
  ]
}
```

**4. Strong-signal gating** — qmd skips expensive LLM query expansion when BM25 top-hit scores ≥0.85 with a ≥0.15 gap to runner-up. Cheap check that saves ~2s on high-confidence lookups. Worth borrowing as a pattern for any tiered-cost pipeline: fast path when signal is strong, fall back to expensive path only when needed.

**5. Chunk-first reranking** — qmd reranks the best *chunk* per document (not the full document). This means the reranker sees ~900 tokens of signal, not 50k. Keeps rerank latency constant regardless of document size. If zazigv2 ever builds its own chunked search, steal this boundary.

## We Do Better

- **Python-native core** — zazig's agent loop has no JS/Bun/Node overhead. qmd requires a separate runtime.
- **Explicit write path** — `MemoryWriter` writes dated markdown files with structured categories. qmd has no concept of writing memories; it only indexes what's already on disk.
- **Multi-agent coordination** — Multiple zazig agents share one qmd index. qmd was designed as a single-user CLI; the HTTP daemon makes sharing safe but qmd has no concept of multi-tenant access or agent-scoped searches.

## Architecture Observations

qmd's three-layer architecture is elegant: `store.ts` owns all database and search logic (BM25, vector, RRF, rerank), `llm.ts` owns model lifecycle (lazy loading, context pools, session ref-counting, inactivity unloading), `mcp.ts` owns protocol translation (stdio + HTTP, dynamic server instructions). The layers are clean — `store.ts` calls into `llm.ts` only via `ILLMSession` interface; `mcp.ts` calls into `store.ts` via exported functions. Swapping the LLM backend would only require changing `llm.ts`.

The `withLLMSession` pattern is notable: it wraps LLM operations with ref-counting and abort signals. When the inactivity timer fires, it checks session ref count before unloading. This prevents VRAM thrashing without explicit locks. Reusable outside of qmd.

The Bun/Node compatibility shim in `db.ts` (dynamic string import to prevent TypeScript from resolving `bun:sqlite` at compile time) is a clever trick worth noting if zazigv2 ever needs to support multiple runtimes.

## Codex Second Opinion

**Model:** gpt-5.3-codex (xhigh reasoning)

**Agrees:** HTTP daemon is the right call. Don't import `store.ts` directly — qmd publishes `dist/` artifacts, not `src/` as a public contract. Importing internals is brittle across updates. Also, you'd inherit native deps in-process (`node-llama-cpp`, `sqlite-vec`, Node ≥22) and all lifecycle sharp edges.

**Differs:** Move context tree higher (above structured multi-query). "Quality per effort is excellent."

**Surfaces:** Chunk-first reranking and strong-signal gating as patterns I underranked. Both are efficiency wins for any search pipeline, not just qmd-specific.

**Final priority from synthesis:** daemon → context tree → structured multi-query → strong-signal gating → session pattern.

## Raw Notes

- Default models: embeddinggemma-300M (embedding), qwen3-reranker-0.6B (rerank), custom fine-tuned qmd-query-expansion-1.7B (query expansion). All GGUF, auto-downloaded from HuggingFace.
- Index lives at `~/.cache/qmd/index.sqlite`. Collections config at `~/.config/qmd/index.yml`.
- HTTP transport: `POST /mcp` (MCP protocol), `POST /query` (plain REST), `GET /health`. Only binds to localhost.
- Etag-based model freshness: HEAD to HuggingFace, store etag file, re-download only on change. Good pattern if we ever manage local models.
- Smart chunking: 900 tokens/chunk, 15% overlap, preference for heading boundaries (h1=100, h2=90, code fence=80, paragraph=20) with squared-distance decay. Never splits inside code fences.
- Grammar-constrained query expansion: BNF grammar forces `lex:/vec:/hyde:` output format, no post-processing needed. Requires a fine-tuned 1.7B model though.
- `.claude-plugin/` directory contains a Claude Code marketplace plugin — qmd ships its own CLAUDE.md and can be installed via `claude marketplace add tobi/qmd`.
- qmd is by Tobi Lütke (Shopify CEO) and is actively maintained (v1.0.8, extensive CHANGELOG, fine-tuned models).
