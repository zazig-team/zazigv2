Org Model / Knowledge

Knowledge System

Two knowledge systems — Doctrines (what agents believe) and Canons (what agents have studied) — because professional judgment is architecturally different from reference material.

The Triad

Three systems work together to make an agent competent. Each serves a distinct purpose in how agents approach their work.

Layer 3
Skills
Teach how to work
Layer 4
Doctrines
Teach what they believe
Layer 5
Canons
Teach what they've studied

Canons inform Doctrines which guide how agents apply Skills. A CTO's doctrine on “always prefer boring technology” is informed by canon knowledge of specific technologies, and guides how they apply their architecture review skill.

Two Systems Comparison

Doctrines and Canons look similar on the surface but differ in almost every dimension. This table maps the key differences.

Doctrines Canons
What they are Role-specific beliefs, heuristics, frameworks Shared factual reference knowledge
Who owns them Each exec role The company — all execs
Size Small: 10–30 claims per lens, ~200 tokens each Massive: 30 books = millions of tokens
Can contradict? Yes — productively No
Authority weights Yes — per-domain epistemic authority No
Human QA On the OUTPUT (founder reviews claims) On the INPUT (founder approves source)
Proactive injection Heavy (~1000–2000 tokens) Light (~100–400 tokens)
Retrieval Mostly proactive, some reactive Mostly reactive, minimal proactive

Progressive Disclosure

Both systems use tiered retrieval so agents get the right depth of knowledge at the right time, without blowing the context window.

Doctrine Tiers

Tier 1
Lens Index
~30–50 tokens per lens. Agent knows what it has doctrine in. Always in prompt.
Tier 2
Lens Map
~200–500 tokens. Agent sees the shape of relevant doctrine. Loaded when topic matches.
Tier 3
Claim Bodies
~100–300 tokens each. Deep expertise when needed. Loaded on direct relevance.

Canon Levels

Level 1
Library Index
Which books exist. Minimal tokens. Always available in the static prefix.
Level 2
Section Map
Chapter and section summaries. Loaded when a book is topically relevant.
Level 3
Passages
Specific chunks with citations. Loaded reactively on high-similarity match.

Cache-Optimized Injection

The prompt stack splits knowledge into a static prefix and dynamic suffix. Positions 1–3b are cached across all jobs for the same role. The result: ~10x cost reduction on knowledge injection.

1
Personality prompt — who you are
2
Role prompt — what you do
3
Doctrine Tier 1 index — stable pointers to what you believe
3b
Canon library pointers — stable pointers to what you've studied
cache break line
4
Proactive doctrine claims — task-relevant beliefs (~1000–2000 tokens)
5
Doctrine tension blocks — cross-role contradictions
6
Canon source summaries — high-similarity passages (~100–400 tokens)
7–9
Skills, task context, memory

The static prefix (positions 1–3b) is identical across all jobs for the same role within a company. Only the dynamic suffix changes per task. This makes prompt caching extremely effective.

Productive Contradictions

Doctrines can — and should — contradict across roles. When the orchestrator detects a cross-role tension relevant to the current task, it injects both positions into the agent's prompt as a tension block. The agent reasons through both.

CTO Position High Authority

“3-week security review needed before launch. Any service handling PII must pass penetration testing and threat modeling before production deployment.”

CPO Position Low Authority on Security

“Ship with basic auth and iterate. We can add hardened security in the next sprint — the market window matters more than perfection.”

The agent receiving this tension block sees both positions with their authority weights. It doesn't blindly follow the highest authority — it reasons through the trade-offs and explains its recommendation. This is how real executive teams work.
