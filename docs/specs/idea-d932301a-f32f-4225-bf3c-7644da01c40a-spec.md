# Spec: Auto-Enrich Phase 2 — Triage Title Discipline + Orchestrator Self-Healing

**Idea ID:** d932301a-f32f-4225-bf3c-7644da01c40a
**Status:** developing
**Complexity:** medium
**Priority:** medium
**Depends on:** Phase 1 manual Enrich button (shipped — `requestEnrichmentJob` in `queries.ts`, Enrich button in `Ideas.tsx`)

---

## Overview

Phase 2 of idea enrichment closes the loop on incomplete triage output without human intervention. It has two components:

1. **Triage skill title discipline** — Phase 2 of the triage skill must always write a clean, intent-capturing title (max 80 chars) on every triage pass, not just in Enrich Mode.
2. **Orchestrator auto-enrich watcher** — a new orchestrator step detects triaged ideas still missing `title` or `description` after a configurable delay and auto-commissions a triage-analyst enrichment job.

Together these ensure that once an idea is triaged, it has complete metadata within minutes — without a human needing to click Enrich.

---

## Background

### Phase 1 (already shipped)
- `requestEnrichmentJob` in `packages/webui/src/lib/queries.ts:1100` dispatches a headless triage-analyst expert session with `action: enrich` context and a list of missing fields.
- The Enrich button in `packages/webui/src/pages/Ideas.tsx:592` is visible when readiness checks find missing fields and calls `handleEnrich`.
- The triage skill (`projects/skills/triage.md`) handles Enrich Mode: parses `{"idea_id", "action": "enrich", "missing": [...]}` and fills only the missing fields via `update_idea`.

### Current gap
- The triage skill Phase 2 does not explicitly call `update_idea` with a clean title on every pass — title-setting is only in Enrich Mode.
- The orchestrator has no step to watch for post-triage ideas with incomplete metadata and auto-heal them.

## Component 1: Triage Skill — Enforce Title on Every Phase 2 Pass

### File
`projects/skills/triage.md`

### Change
Add an explicit title-writing step to Phase 2 after the deduplication check, before setting priority:

> **1b. Set a clean title (mandatory):**
> Generate or confirm a `title` (max 80 chars) that captures the core intent of the idea. The title must be distinct from `raw_text` — it should be a semantic summary, not a truncation. If the existing title already meets this standard, keep it. Otherwise, generate a new one and include it in the `update_idea` call in step 2.

The existing Phase 2 step 2 (`update_idea` to refine description and add/correct tags) is extended to also always include `title` in the payload.

### Rules
- Title must be ≤ 80 characters
- Title must capture the core intent, not be a raw truncation
- Title must be written on every Phase 2 pass (auto-triage and single-idea mode)
- If the existing title already meets the standard, re-use it (no unnecessary mutation)

## Component 2: Orchestrator — Auto-Enrich Watcher

### File
`supabase/functions/orchestrator/index.ts`

### New function: `autoEnrichIncompleteTriagedIdeas`

Called in the main orchestrator tick, after `autoSpecTriagedIdeas`.

**Logic:**

1. Query ideas where:
   - `status = 'triaged'`
   - `triage_route = 'develop'` (or any non-parked/rejected route — configurable)
   - `(title IS NULL OR title = '' OR description IS NULL OR description = '')`
   - `triaged_at < now() - interval '<enrich_delay_minutes> minutes'`
   - Company has `auto_triage = true` (reuse the same gate as auto-triage; add `auto_enrich` flag optionally, or piggyback on `auto_triage`)
2. For each such idea, check there is no active (status in `queued`, `executing`) job for that idea with role `triage-analyst` — to avoid double-enriching.
3. Dispatch a headless triage-analyst expert session via `POST /functions/v1/start-expert-session` with brief:
   ```
   Enrich idea {idea_id}. Fill in missing fields: {missing_fields_list}. Call update_idea with the enriched data.
   ```
   This is identical to what `requestEnrichmentJob` in the webui does — the orchestrator reuses the same brief format.
4. Log each dispatch: `[orchestrator] Auto-enriching idea {id} — missing: {fields}`.

### Company settings
- Reuse `auto_triage = true` as the gate (no new column required for MVP).
- Add `enrich_delay_minutes` company column (default: 10) via a new migration — this controls how long to wait after triage before auto-enriching. Keeps it configurable without being a magic number.
- Max concurrent enrichment jobs: reuse the existing `triage_max_concurrent` limit or add `enrich_max_concurrent` (default: 2).

### Migration: `NNN_company_enrich_settings.sql`
```sql
ALTER TABLE companies
  ADD COLUMN IF NOT EXISTS enrich_delay_minutes integer NOT NULL DEFAULT 10,
  ADD COLUMN IF NOT EXISTS enrich_max_concurrent integer NOT NULL DEFAULT 2;
```

### Idempotency
- The function checks for an active triage-analyst job scoped to the idea before dispatching.
- If the idea is already being enriched, skip it (same pattern as `autoSpecTriagedIdeas`).

## Acceptance Criteria

1. After the triage skill runs Phase 2 on any idea, the idea's `title` field in the database is non-null, ≤ 80 characters, and semantically describes the core intent (not a raw truncation of `raw_text`).
2. An idea that reaches `status='triaged'` with a null or empty `title` or `description` is automatically enriched (by the orchestrator) within `enrich_delay_minutes` + one orchestrator tick (~10 seconds).
3. Auto-enrichment is not dispatched if a triage-analyst enrichment job for that idea is already active (queued or executing).
4. Auto-enrichment respects the `auto_triage = true` company gate — ideas for companies with `auto_triage = false` are not auto-enriched.
5. After auto-enrichment completes, the idea's `title` and `description` are populated and the Enrich button no longer appears in the WebUI for that idea.
6. The orchestrator logs a `[orchestrator] Auto-enriching idea {id}` message for each dispatched enrichment job.
7. Existing behavior of the Enrich button (Phase 1) is unaffected.
8. The triage skill's Enrich Mode continues to work correctly when triggered manually or by the orchestrator.
