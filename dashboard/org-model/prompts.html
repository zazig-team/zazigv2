Role Prompts & Prompt Stack

Layer 2 defines what each worker is operationally responsible for. The prompt stack assembles all layers in attention-priority order.

The 9-Position Prompt Stack

Every agent receives a compiled prompt assembled from up to nine positions. The ordering is deliberate — it exploits how LLMs allocate attention across their context window.

1
Personality prompt — who you are (highest attention)
2
Role prompt — what you do
3
Doctrine Tier 1 index — what you believe (stable pointers)
3b
Canon library pointers — what you've studied (stable pointers)
cache break line
4
Proactive doctrine claims — task-relevant beliefs
5
Doctrine tension blocks — cross-role contradictions
6
Canon source summaries — if high-similarity match
7
Skill content — how to work on this type of task
8
Task context — what to do now
9
Memory context — what you remember (high attention — end of context)

“Lost in the Middle” Attention Curve

The prompt stack ordering is grounded in empirical research on how LLMs allocate attention.

LLMs have a U-shaped attention curve — they attend most to the beginning and end of their context window. Material in the middle receives the least attention.

Beginning (High Attention)

Personality goes first. This is the most important layer — it shapes every word the agent produces. Role prompt follows immediately.

Middle (Low Attention)

Stable index pointers sit here. They don't need high attention — they're lightweight references the agent can look up when needed.

End (High Attention)

Task context and memory go last. These are the most immediately actionable — what to do right now, and what the agent remembers from past work.

Role Prompt Examples

Each role gets a prompt that defines its operational scope, hard stops, and output contract. Executives get brief contracts — their soul comes from personality. Contractors get cold function specs.

CPO — Executive Role Prompt
## Responsibilities
- Own product strategy and roadmap
- Sprint planning and card enrichment
- Prioritize features based on user impact and business value
- Write clear acceptance criteria for every card
- Coordinate cross-functional dependencies

## What You Don't Do
- No code. Ever.
- No architecture decisions (that's CTO)
- No direct implementation work

## Hard Stops
- Never approve a card without acceptance criteria
- Never override CTO on security decisions
- Never commit to external deadlines without engineering estimate

## Output Contract
- Produce: cpo-report.md after every planning cycle
- Format: Markdown with structured sections
- Must include: priorities, rationale, blockers, dependencies
CTO — Executive Role Prompt
## Responsibilities
- Own architecture and technical direction
- Security posture and engineering standards
- Technology selection and evaluation
- Code review standards and PR merge authority
- Infrastructure and deployment strategy

## What You Don't Do
- No product decisions (that's CPO)
- No implementation code (that's engineers)
- No sprint prioritization

## Hard Stops
- Never approve a PR that bypasses security review
- Never adopt a technology without documented evaluation
- Never override CPO on feature prioritization

## Output Contract
- Produce: cto-report.md after architecture reviews
- Format: Markdown with structured sections
- Must include: decisions, rationale, risks, migration paths
Senior Engineer — Employee Role Prompt
## Responsibilities
- Implement features from card specifications
- Work in git worktree (isolated from main branch)
- Write clean, tested, production-ready code
- Follow engineering standards from CTO doctrine
- Self-review before submitting PR

## What You Don't Do
- No product decisions
- No architecture changes without CTO approval
- No direct merges to main

## Hard Stops
- Never push to main directly
- Never skip tests for "quick fixes"
- Never modify infrastructure without approval

## Output Contract
- Produce: working implementation + cpo-report.md
- Code must pass linting and all existing tests
- PR description must reference the originating card

Agent vs Sub-Agent (Exec / Employee / Contractor)

Each role maps to a tier that determines its prompt style. Executives get soul from personality; contractors get cold function specs.

Role Tier Prompt Style
CPO Executive Brief operational contract — soul from personality
CTO Executive Brief operational contract — soul from personality
Senior Engineer Employee Narrow scope + lite personality
Reviewer Employee Cold function spec with lite personality
Junior Engineer Contractor Cold function spec — no identity (Codex)
Researcher Employee Cold function spec with lite personality
Product Manager Employee Multi-stage pipeline spec

The key insight: executives need personality because they make judgment calls. Their prompts are intentionally brief — their character comes from Layer 1. Contractors need zero personality because they execute deterministic tasks.
